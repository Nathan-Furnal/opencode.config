# Principles

- Immutable data structures over mutable ones.
- Avoid nullable types unless nullability has specific semantics.
- Parse, don't validate.
- Prefer composition over inheritance unless tracking upstream behavior.
- Pure functions preferred. Side effects isolated at boundaries.
- Explicit error types or exceptions. No swallowing errors silently.
- Read existing code before modifying. Match existing patterns and conventions.
- Comments should stay in sync with the code and describe why it does something.
- For structural forks (module boundaries, when to add indirection, when to
  refactor), load the `design-principles` skill rather than deciding ad hoc.

## Verification

- Verify changes compile/pass before moving to the next task.
- When writing new functionality, write a failing test first, then implement.
- Do not mark work as done without running verification.

## Workflow

- Commit after each logical change. Descriptive message, imperative mood.
- For multi-file work, decompose into independently-verifiable steps.
- When uncertain about architecture, use @explore or Plan mode before editing.
- When stuck, explain what you tried and what failed before retrying.

## Session Memory

- Session summary is auto-generated by the harness plugin. Do not write it manually.
- At session start for continuing work, read `.opencode/memory/session-summary.md` before
  beginning, and call `memory_recall` to reload durable notes about the area you're
  touching. Use `memory_retain` to save decisions, gotchas, and invariants worth keeping.

## Spec Format

- Specs are assertion checklists, not prose documents. Each line is one testable claim.
- Spec changes are append-only deltas logged with a version marker.
- When refining specs, add/remove/change individual assertions. Do not rewrite the
  document.
